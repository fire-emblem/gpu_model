# Inst Decode And Disassemble Framework

## Objective

Move device-side ingestion from:

- `objdump` text
- ad-hoc text lowering
- execution on canonical asm only

toward:

- ELF code object
- code-section bytes
- instruction decode from encoding definitions
- disassembly generated by the project
- execution on decoded raw instructions

The immediate goal is not to implement every GCN encoding at once.
The goal is to define a framework that makes new encodings, new decoders, and new
disassemblers append-only through configuration lists.

## Pipeline Shape

Target steady-state pipeline:

1. ELF / code object reader
2. section extraction
3. raw instruction decode
4. raw instruction IR
5. project disassembler
6. lowering or direct execution

Current transitional pipeline:

1. ELF / code object reader
2. section extraction
3. raw instruction container from code object text decode helper
4. text-based lowering or execution

## Layer Split

### 1. Code Object Layer

Purpose:

- load binary sections, metadata, and symbols from an AMDGPU code object

Suggested types:

- `AmdgpuCodeObjectImage`
- `CodeSectionView`
- `KernelSymbolView`
- `KernelMetadataView`

Current implementation direction:

- [amdgpu_code_object_decoder.h](/data/gpu_model/include/gpu_model/loader/amdgpu_code_object_decoder.h)
- [amdgpu_code_object_decoder.cpp](/data/gpu_model/src/loader/amdgpu_code_object_decoder.cpp)

### 2. Encoding Definition Layer

Purpose:

- define instruction encodings declaratively
- avoid opcode-specific decode branches scattered across the project

Suggested core structure:

- `InstEncodingDef`

Suggested fields:

- ISA family
- mnemonic
- major encoding class:
  - `SOP1`
  - `SOP2`
  - `SOPC`
  - `SOPK`
  - `SOPP`
  - `SMEM`
  - `VOP1`
  - `VOP2`
  - `VOP3`
  - `VOPC`
  - `DS`
  - `MUBUF`
  - `FLAT`
- instruction size:
  - 4 bytes
  - 8 bytes
- bitfield layout
- operand decode recipe
- semantic family
- optional canonical lowering recipe

Design rule:

- new instructions should be added by appending entries to encoding-definition lists
- not by growing central decode `switch` logic

### 3. Raw Decode Layer

Purpose:

- read bytes and instantiate decoded instruction objects

Suggested interfaces:

- `IInstDecoder`
- `InstDecoderRegistry`

Suggested decode input:

- `std::span<const std::byte>`
- current PC
- target ISA / gfx level

Suggested decode output:

- `RawDecodedInstruction`

Suggested fields:

- PC
- size
- matched encoding id
- raw words
- decoded operands
- decode flags

### 4. Raw Instruction IR

Purpose:

- be the project-owned result of binary decode
- independent from textual syntax

Current seed type:

- [raw_gcn_instruction.h](/data/gpu_model/include/gpu_model/instruction/encoded/encoded_gcn_instruction.h)

Recommended evolution:

- keep raw words
- add encoding class id
- add decoded operand list
- add special-register references
- add register-range operands
- add flags such as:
  - writes `vcc`
  - writes `exec`
  - scalar memory
  - vector memory

### 5. Disassembler Layer

Purpose:

- format decoded instructions back to stable textual assembly
- use project decode results instead of external `objdump`

Suggested interfaces:

- `IInstFormatter`
- `InstFormatterRegistry`

Design rule:

- formatting should use the decoded instruction object plus encoding definition
- not a second independent parser or mnemonic-specific pretty-printer chain

### 6. Execution Bridge Layer

Purpose:

- choose between:
  - direct raw execution
  - lowering to canonical internal ops

Near-term recommendation:

- use raw decode as the source of truth
- still allow a lowering bridge for partially supported instructions
- make unsupported raw instructions fail with explicit diagnostics

Long-term recommendation:

- execute decoded raw instructions directly through raw semantic handlers

## Configuration-List Design

All extension seams should follow the same shape:

- descriptor list
- registry lookup
- derived handler or formatter class

Apply this to:

- code object decoders
- instruction decoders
- disassemblers
- lowering rules
- semantic handlers

Avoid:

- one huge decoder `switch`
- one huge disassembler `switch`
- one huge “if mnemonic == ...” formatter block

## Minimal GCN Decode Rollout

### Phase 1

Implement encoding definitions and raw decode for the minimum vecadd path:

- `s_load_dword`
- `s_load_dwordx2`
- `s_load_dwordx4`
- `s_waitcnt`
- `s_and_b32`
- `s_mul_i32`
- `v_add_u32_e32`
- `v_cmp_gt_i32_e32`
- `s_and_saveexec_b64`
- `s_cbranch_execz`
- `v_ashrrev_i32_e32`
- `v_lshlrev_b64`
- `v_mov_b32_e32`
- `v_add_co_u32_e32`
- `v_addc_co_u32_e32`
- `global_load_dword`
- `v_add_f32_e32`
- `global_store_dword`
- `s_endpgm`

### Phase 2

Use project disassembler to render decoded raw instructions and compare against
`llvm-objdump` for golden validation.

### Phase 3

Route `.out` host-runtime interception to:

- code object decode
- raw instruction IR
- raw execution or raw-to-canonical lowering

instead of current text-based lowering.

## Testing Strategy

### Decoder Tests

- given code bytes, decode expected raw instruction fields
- verify instruction length and operand extraction

### Disassembler Tests

- given decoded instruction, emit expected AMD-style text

### Round-Trip Tests

- code object -> decode -> disassemble
- compare against stable golden text

### End-to-End Tests

- real HIP `.out`
- host `main()` native execution
- runtime hook launch
- raw code object decode
- model kernel execution
- host-side result validation

## Immediate Next Step

The next implementation step should be:

1. add `InstEncodingDef`
2. add `RawDecodedInstruction`
3. add a minimal `GcnInstDecoder` for the Phase 1 vecadd opcodes
4. add a `EncodedInstructionFormatter`
5. keep the current text path as fallback until raw decode coverage is sufficient
