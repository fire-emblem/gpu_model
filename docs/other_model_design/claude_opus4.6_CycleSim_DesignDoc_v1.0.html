1. 概述与设计目标

1.1 项目背景

在 AMDGPU（GCN / CDNA / RDNA）架构上进行编译器后端优化和高性能算子库开发时，工程师需要深入理解指令调度、寄存器分配、内存访问模式等对实际执行 cycle 数的影响。现有工具存在两个极端：一端是硬件 RTL 仿真（Verilog 级模拟），cycle 精确但极其缓慢，单个 kernel 可能需要数小时甚至数天完成；另一端是 rocprof 等运行时 profiler，能给出宏观统计数据（如总耗时、L2 miss 率等），但无法提供指令级 cycle 归因分析和 "what-if" 假设验证能力。

本项目旨在填补这一空白，构建一个轻量级、指令精确（instruction-accurate）且 cycle 近似（cycle-approximate）的纯软件模拟器。使编译器和算子开发者能够：在无需真实 GPU 硬件的环境下快速迭代优化方案；获得指令粒度的 cycle 归因信息；通过可视化时间线直观观察流水线气泡和资源冲突；验证算法修改对整体 kernel 运行时间的影响。

1.2 核心目标

目标一：功能正确性。对于任意合法的 HIP 程序（编译后的 .out 可执行文件），模拟器拦截全部 GPU kernel 执行，在 host 侧以软件方式完成全部 GPU 计算。Host 程序获得与真实 GPU 执行完全一致的数值结果，能够通过端到端校验。这是所有后续 cycle 分析的前提基础。

目标二：全程序 Cycle 估算。以 Compute Unit（CU）为基本模拟粒度，建模指令流水线、功能部件占用、数据依赖 stall、内存延迟、wave 调度仲裁和 barrier 同步，输出整个 kernel（乃至包含多次 kernel launch 的完整程序）的总 cycle 估算值。输出的不是孤立的单 wave cycle，而是考虑了多 wave 并发、多 workgroup 在多 CU 上调度后的全局 cycle。

目标三：气泡与瓶颈归因。自动识别并量化流水线中的气泡（bubble）来源：计算受限（compute bound，功能部件利用率饱和）、访存受限（memory bound，VMEM/SMEM 延迟未被隐藏）、延迟受限（latency bound，occupancy 不足无法切换 wave 隐藏延迟）、LDS bank conflict、barrier 等待、分支发散等。以部件为粒度统计空闲 cycle 占比。

目标四：指令级 Trace 与 Log。提供可配置的详细 trace 输出：每条指令执行时的 PC 地址、当前 cycle、wave ID、指令助记符、源/目标寄存器值（或摘要）、stall 原因等。支持按 wave 过滤，支持 binary trace 和 human-readable text 两种格式。

目标五：Perfetto 时间线可视化。生成 Perfetto 兼容的 protobuf trace 文件，在 Perfetto UI (ui.perfetto.dev) 中以多 track 形式展示：每个 CU 的每个 SIMD lane 上 wave 的指令发射时间线、stall 着色标注、barrier 同步点、kernel 起止边界等。

1.3 设计原则与约束

轻量优先。整个模拟器为纯 C++ 软件实现（辅以 Python 脚本工具），无需 GPU 硬件，可在普通 x86 Linux 服务器或笔记本上运行。对于典型算子 kernel（数千条指令、数百个 wave），单次模拟应在秒到分钟量级完成。绝不追求 RTL 级精度，而是在"能够正确反映相对性能趋势"的前提下最大化模拟速度。

可配置性。微架构参数（CU 数量、每 CU 的 SIMD 数、VGPR/SGPR 总量、各级缓存容量与延迟、HBM 带宽等）均通过 JSON/YAML 配置文件指定。内置 MI100（gfx908）、MI210（gfx90a）、MI300X（gfx940/942）等预设配置，也允许用户自定义参数以模拟假想架构或探索设计空间。

渐进精度。系统采用分层架构。功能仿真层保证正确性且独立于 cycle 模型；cycle 模型层可独立替换或升级。初期使用表驱动的静态延迟模型，后续可逐步引入 cache miss 概率模型、bank conflict 精确建模等，而不影响功能正确性。

透明接入。用户无需修改源代码或重新编译 kernel。通过 LD_PRELOAD=libgpusim_hook.so ./my_program.out 即可自动进入模拟模式。

不做的事。不模拟 GPU 图形流水线（仅 compute 路径）；不模拟 PCIe 传输延迟（hipMemcpy 做功能模拟但不计时）；不模拟多 GPU；不模拟 host-device 并发重叠（kernel 视为同步执行）。

2.2 模块划分与职责

Runtime 拦截层（libgpusim_hook.so）是用户接触模拟器的唯一入口。它以 LD_PRELOAD 方式注入进程空间，通过符号劫持覆盖 HIP runtime 库的关键 API。hipMalloc 被重定向为 host 侧 mmap 分配（维护一张 device 虚拟地址→host 实际地址的映射表），hipMemcpy 变为 host 侧 memcpy，hipLaunchKernel 将 kernel 启动请求转交模拟引擎，hipDeviceSynchronize 等待模拟完成。对于 host 侧的纯 CPU 逻辑（数据预处理、结果读回校验等），进程照常在真实 CPU 上执行，保证端到端语义不变。

ELF / Code Object 解析器负责从 .out 中嵌入的 AMDGPU Code Object（fatbin 格式）中提取 kernel 二进制。Code Object 本质是 ELF 文件，解析器读取 .text（机器码）、.rodata（常量数据）、AMDGPU metadata（YAML/MsgPack 格式的 kernel 描述，含 VGPR/SGPR 用量、LDS 大小、workgroup size 限制等）、kernel descriptor（含 program 入口偏移、各种 enable 标志等）。支持 Code Object V3 及以上版本。

指令解码引擎将 .text 段的二进制指令流解码为结构化的指令对象数组。每个 InsnObj 包含：PC 偏移、操作码枚举、指令类型分类（VALU/SALU/VMEM/SMEM/LDS/FLAT/BRANCH/MFMA/EXPORT 等）、源操作数列表、目标操作数列表、立即数、指令长度（4 或 8 字节）、以及通过查表获得的 timing 属性（issue latency、execute latency）。解码器参考 LLVM 的 AMDGPU 后端 TableGen 定义来构建解码表，保证指令集覆盖完整。

功能仿真核心（FuncSim）是保证数值正确性的关键模块。它按 wave 为单位执行指令序列：维护每个 wave 的 VGPR 文件（64 lanes × N 个 VGPR）、SGPR 文件、EXEC mask、VCC、PC 等状态。每条指令的 execute() 方法按照 ISA 手册语义修改寄存器状态。对于 VMEM load/store，访问 UMM 提供的 host 侧内存映射；对于 LDS 操作，访问 workgroup 粒度分配的 LDS buffer；对于分支指令，修改 PC 和 EXEC mask。功能仿真与 cycle 模型完全解耦——即使不跑 cycle 模型，也能得到完全正确的计算结果。

CU 微架构 Cycle 模型不进入 RTL 级别的流水线建模，而是采用"发射 slot + 延迟表 + scoreboard"的抽象模型。每个 CU 维护若干功能部件的"最早可用 cycle"状态，以及每个 VGPR/SGPR 的"数据就绪 cycle"（scoreboard）。指令发射时检查：源操作数是否就绪、目标功能部件是否空闲、是否有 waitcnt 限制等。若不满足则产生 stall bubble 并记录原因。发射后更新目标寄存器的就绪时间和功能部件占用。这一模型足以捕捉数据依赖 stall、功能部件冲突、内存延迟暴露等关键性能特征。

Wave 调度器模型模拟 CU 内部的 wave 调度行为。AMDGPU 硬件中每个 SIMD 采用轮转（round-robin）或最旧优先（oldest-first）策略在多个 wave 间切换发射。模拟器在每个模拟 cycle 中为每个 SIMD 选择一个 ready wave 发射一条指令（或记录一个 bubble）。长延迟操作（VMEM、SMEM）可使 wave 进入 wait 状态，调度器将其跳过直至数据返回。

内存子系统模型采用简化的分层延迟模型。VMEM 请求先查 L1 vCache（配置化延迟，如命中 ~40 cycle），miss 则访问 L2（~120 cycle），再 miss 访问 HBM（~300 cycle）。初期可使用固定延迟（配置文件中指定），后期可引入概率化 cache 模型。SMEM 类似但走标量缓存路径。LDS 访问通常为 ~20 cycle，bank conflict 可选建模。带宽限制以简化方式建模：统计单位时间内的 outstanding 请求数，当超过带宽上限时对新请求施加额外延迟。

Grid 级调度器模拟 Command Processor 将 workgroup 分配到 CU 的过程。依据每个 CU 的资源余量（VGPR、SGPR、LDS、wave slot），决定新 workgroup 是否可以分配到某个 CU。当一个 workgroup 内所有 wave 执行完毕后释放资源，允许新 workgroup 入驻。这一层决定了 occupancy 和全局并行度。

Log / Trace 子系统负责在模拟过程中记录指令级事件。支持三种粒度：关闭（仅输出最终统计）、摘要模式（每条指令记录 PC + cycle + stall 信息）、详细模式（额外记录所有寄存器读写值）。输出格式支持文本（grep 友好的列式格式）和二进制（紧凑高效，附配套解析工具）。

性能分析引擎在模拟完成后汇总全部事件数据，输出多维度统计报告和 Perfetto trace 文件。

2.4 精度与轻量化权衡

模拟器的核心权衡在于"模拟速度"与"cycle 精度"之间的平衡。我们采用以下策略实现轻量化。

表驱动延迟而非流水线微码。不建模指令在流水线中逐级推进的过程，而是为每种指令类型维护一张 (issue_latency, execute_latency) 查找表。issue_latency 表示该指令占用发射端口的 cycle 数（通常为 1），execute_latency 表示结果可被读取的延迟。这足以捕捉依赖链上的关键路径长度，而无需追踪流水线每一级。

简化的 cache 模型。初始版本不模拟精确的 cache 行为（set-associative tag 匹配等），而是使用可配置的固定延迟或简单概率模型（如 "L1 命中率 80%、L2 命中率 95%"）。对于需要精确 cache 分析的场景，预留接口供后续接入更精确的 cache 模拟模块。

功能仿真与 cycle 模型分离。功能仿真按指令顺序执行，不受 cycle 模型影响。cycle 模型作为"旁路观察者"，在功能执行的同时进行 timing 记账。这避免了 cycle 模型 bug 影响功能正确性，也使得两个模块可以独立开发和测试。

可选的模拟范围。用户可以选择只模拟前 N 个 workgroup、或只模拟特定 CU 上的行为，以缩短大规模 kernel 的模拟时间。提供 "采样模式" 用少量代表性 workgroup 估算整体性能。

2.5 输出体系总览

输出体系分为四大类。指令级 Trace Log 是最详细的输出，逐指令记录执行轨迹。整体统计报告 汇总全 kernel（或全程序多 kernel）的宏观性能数据。Perfetto Trace 是用于可视化的时间线文件。终端摘要 是运行结束时打印到标准输出的简洁信息。具体格式和内容在第 6 章详述。

3. Low Level 设计

3.1 Runtime 拦截层

3.1.1 拦截机制

采用 LD_PRELOAD + dlsym(RTLD_NEXT, ...) 模式。编译为共享库 libgpusim_hook.so，其中定义与 HIP Runtime API 同名的函数符号。当用户程序调用 hipMalloc 时，链接器优先解析到我们的符号。若某些 API 不需要拦截（如 hipGetDeviceProperties 等查询类），通过 dlsym(RTLD_NEXT, "hipXxx") 透传给真实 HIP Runtime（若存在），或返回预设的静态假数据。

3.1.2 拦截的 API 清单

API拦截行为
hipMalloc在 host 侧 mmap 分配等大小内存，将返回的 host 指针记录在地址映射表中，向用户返回伪 device 指针（可以直接用 host 指针，因为功能仿真就在 host 执行）
hipFree释放对应的 host 内存，从映射表中删除
hipMemcpy (H2D)host → 模拟 device 内存的 memcpy
hipMemcpy (D2H)模拟 device 内存 → host 的 memcpy
hipMemcpy (D2D)模拟 device 内存之间的 memcpy
hipMemset对模拟 device 内存执行 memset
hipLaunchKernel核心拦截点。提取 kernel function handle、grid/block dim、shared mem size、kernel args，转交模拟引擎执行
hipDeviceSynchronize等待模拟引擎中所有 pending kernel 完成（当前实现为同步模式，launch 即执行完毕）
hipGetDeviceProperties返回配置文件中定义的模拟 GPU 属性
hipStreamCreate / Destroy记录 stream 但当前不模拟多 stream 并发

3.1.3 Kernel Handle 解析

HIP 编译器将 __global__ 函数编译后，在 host 代码中通过 __hipRegisterFunction 注册 kernel stub。我们同样拦截此注册函数，记录 kernel name → device function 映射。当 hipLaunchKernel 传入 function pointer 时，通过映射表找到对应的 kernel name，再从 fatbin 中解析出对应的 Code Object 和 .text 段。fatbin 嵌入在 ELF 的 .hip_fatbin section 中，首次使用时扫描解析并缓存。

3.2 ELF / Code Object 解析器

3.2.1 解析流程

AMDGPU Code Object 嵌入在 host ELF 中。解析分三步：首先，打开 .out 文件，读取 host ELF section headers，找到 .hip_fatbin section（或通过 __hipFatBinSegment 全局变量定位 bundled binary）。其次，fatbin 内部包含多个 offload bundle（按 GPU target 索引），根据配置文件中指定的 target triple（如 amdgcn-amd-amdhsa--gfx90a）选择匹配的 bundle。最后，每个 bundle 就是一个 AMDGPU ELF，使用标准 ELF 解析库（如 libelf 或自写轻量 parser）提取以下信息。

3.2.2 提取信息

信息来源用途
Kernel 机器码.text section 中对应 kernel symbol 的字节流送入指令解码引擎
Kernel Descriptor.rodata 中的 kernel_descriptor 结构（64 字节）获取 VGPR/SGPR enable 数量、LDS size、wave size 等
AMDGPU Metadata.note section（MsgPack 格式或 YAML）kernel name、参数布局、resource usage、workgroup size 限制
Symbol Table.symtab / .dynsymkernel 入口地址偏移、全局变量
重定位表.rela.text处理 global/constant 内存地址引用（通常被 relocate 到绝对地址）

3.2.3 Kernel Descriptor 结构

struct KernelDescriptor {
    uint32_t group_segment_fixed_size;   // LDS 大小 (bytes)
    uint32_t private_segment_fixed_size;  // scratch 大小 (bytes per work-item)
    uint64_t kernel_code_entry_byte_offset; // 入口偏移 (相对 descriptor 起始)
    uint32_t compute_pgm_rsrc1;           // VGPR数、SGPR数、float mode 等
    uint32_t compute_pgm_rsrc2;           // LDS大小、user_sgpr 数等
    uint16_t kernel_code_properties;       // enable_sgpr_kernarg_segment_ptr 等
    // ... 其他字段
};

3.3 指令解码引擎

3.3.1 解码表设计

AMDGPU ISA 指令编码分为多种格式：SOP1/SOP2/SOPK/SOPC/SOPP（标量指令）、VOP1/VOP2/VOP3/VOPC（向量指令）、MUBUF/MTBUF/MIMG（向量内存）、SMEM（标量内存）、DS（LDS/GDS）、FLAT/GLOBAL/SCRATCH、VINTRP 等。每种格式有特定的编码布局。解码器通过指令字的高位 bit pattern 判断格式，再根据具体 opcode 查表得到操作码含义。

解码表以静态数组形式组织，由构建脚本从 LLVM AMDGPU TableGen 定义自动生成（或手动维护高频指令子集）。每条表项包含：opcode 枚举、助记符字符串、指令类型分类、操作数格式描述、issue_latency、execute_latency。

3.3.2 InsnObj 结构

struct InsnObj {
    uint32_t pc;              // 指令的字节偏移 (相对 kernel 入口)
    Opcode   opcode;           // 操作码枚举
    InsnType type;             // VALU/SALU/VMEM/SMEM/LDS/FLAT/BRANCH/MFMA/EXPORT/...
    uint8_t  num_src;          // 源操作数个数
    Operand  src[4];           // 源操作数 (寄存器ID/立即数/特殊寄存器)
    uint8_t  num_dst;          // 目标操作数个数
    Operand  dst[2];           // 目标操作数
    uint32_t imm;              // 立即数 (若有)
    uint8_t  size_bytes;       // 指令长度: 4 或 8
    uint16_t issue_latency;    // 发射延迟 (cycles, 通常 1)
    uint16_t exec_latency;     // 执行延迟 (cycles, 结果可读)
    uint16_t waitcnt_flags;    // 对于 s_waitcnt: vmcnt/lgkmcnt/expcnt 值
    const char* mnemonic;     // 助记符字符串指针
};

3.3.3 解码缓存

同一个 kernel 可能被多次 launch（不同 grid size 但相同代码），因此解码结果以 kernel code hash 为 key 缓存在 hash map 中。首次解码后，后续 launch 直接复用 InsnObj 数组，避免重复解码开销。

3.4 功能仿真核心

3.4.1 Wave 状态

struct WaveState {
    uint32_t wave_id;                  // 全局 wave ID
    uint32_t workgroup_id;             // 所属 workgroup ID
    uint32_t simd_id;                  // 所在 SIMD 编号
    uint32_t cu_id;                    // 所在 CU 编号
    uint32_t pc;                       // 当前 Program Counter
    uint64_t exec_mask;                // EXEC mask (64-bit)
    uint32_t vgpr[256][64];            // VGPR 文件: [reg_id][lane], 按需分配
    uint32_t sgpr[108];                // SGPR 文件
    uint64_t vcc;                      // Vector Condition Code
    uint32_t scc;                      // Scalar Condition Code (1-bit)
    uint32_t m0;                       // M0 special register
    WaveStatus status;                 // RUNNING / STALLED / BARRIER_WAIT / FINISHED
    uint32_t barrier_id;               // 当前等待的 barrier slot
    uint32_t vmcnt, lgkmcnt, expcnt;   // waitcnt 计数器
    uint32_t thread_ids[64][3];       // 每个 lane 的 (tid.x, tid.y, tid.z)
    uint32_t workgroup_ids[3];        // (blockIdx.x, y, z)
};

3.4.2 指令执行

功能仿真按指令类型分发到对应的 execute handler 函数。以 v_add_f32 为例：读取 src0（VGPR 或立即数）和 src1，对 EXEC mask 中每个 active lane 执行浮点加法，写入 dst VGPR。对于 VMEM load（如 buffer_load_dword）：根据 VGPR 中的地址（加上 SGPR 中的 base/offset）计算每个 lane 的访存地址，从 host 内存映射中读取数据写入目标 VGPR。分支指令（如 s_cbranch_scc0）根据 SCC 修改 PC。s_barrier 将 wave 状态设为 BARRIER_WAIT。

3.4.3 EXEC Mask 与分支发散

条件向量指令（如 v_cmp_lt_f32）将比较结果写入 VCC 或指定 SGPR pair。后续 s_and_b64 exec, exec, vcc 可缩小 EXEC mask 实现条件执行。s_or_b64 exec, exec, saved_mask 恢复。功能仿真精确维护 EXEC mask，保证分支发散场景下的正确性。

3.5 CU 微架构 Cycle 模型

3.5.1 模型核心思想

不进行 RTL 级流水线仿真，而是采用 "事件驱动 + scoreboard" 的轻量模型。核心数据结构是每个 CU 维护的状态：

struct CUTimingState {
    uint64_t current_cycle;              // CU 当前全局 cycle
    uint64_t valu_ready_cycle[4];         // 每个 SIMD 的 VALU 最早可发射 cycle
    uint64_t salu_ready_cycle;            // SALU 最早可发射 cycle
    uint64_t vmem_ready_cycle;            // VMEM 单元最早可发射 cycle
    uint64_t smem_ready_cycle;            // SMEM 单元最早可发射 cycle
    uint64_t lds_ready_cycle;             // LDS 单元最早可发射 cycle
    uint64_t branch_ready_cycle;          // Branch 单元最早可发射 cycle
    uint64_t mfma_ready_cycle;            // MFMA 单元最早可发射 cycle
    
    // Per-wave scoreboard
    struct WaveScoreboard {
        uint64_t vgpr_ready[256];   // 每个VGPR的数据就绪cycle
        uint64_t sgpr_ready[108];   // 每个SGPR的数据就绪cycle
        uint64_t vcc_ready;          // VCC就绪cycle
        uint64_t scc_ready;          // SCC就绪cycle
    } scoreboard[MAX_WAVES_PER_CU];
    
    // 统计数据 (per CU)
    BubbleStats bubble_stats;
};

3.5.2 指令发射判定

在模拟的每个 cycle 中，对每个 CU 的每个 SIMD 执行如下判定流程。Wave 调度器选出候选 wave（见 3.6 节）。取该 wave 当前 PC 指向的指令 insn。然后执行一系列检查：源操作数检查，遍历 insn 的所有 src operand，若任一 src 寄存器的 scoreboard[wave].reg_ready[src] > current_cycle，则该 wave 因数据依赖 stall（记录 bubble 原因为 DATA_DEPENDENCY）。功能部件检查，根据 insn.type 查对应部件的 ready_cycle，若 > current_cycle，则因部件冲突 stall（记录 UNIT_BUSY）。Waitcnt 检查，若 wave 有 outstanding 的内存请求，且最新 s_waitcnt 要求 vmcnt ≤ N 但当前 outstanding > N，则 stall（记录 WAITCNT）。若所有检查通过，指令成功发射。更新操作包括：将目标寄存器的 scoreboard ready 时间设为 current_cycle + exec_latency；将功能部件的 ready_cycle 设为 current_cycle + issue_latency；递增 wave 的 PC；触发功能仿真执行该指令（更新寄存器值和内存）；记录 trace event。

3.6 Wave 调度器模型

3.6.1 调度策略

每个 SIMD 上可驻留多个 wave（典型为 8~10 个，取决于 VGPR/SGPR 资源压力）。在每个模拟 cycle，调度器需要为每个 SIMD 选择最多一个 wave 发射指令。模拟器支持多种可配置的调度策略。

Round-Robin 是默认策略，按固定顺序轮转选择 ready 状态的 wave。这是 AMDGPU 硬件的近似行为。Oldest-First 优先选择等待时间最长（距离上次发射 cycle 最远）的 wave。Greedy (Youngest-First) 优先选择最近刚准备好的 wave，倾向于让单个 wave 快速推进。用户可通过配置文件切换策略，观察不同策略对性能的影响。

3.6.2 Wave 状态机

图 3-1：Wave 状态机

3.6.3 Occupancy 与 Wave Slot 管理

每个 CU 的 wave slot 总数由 VGPR、SGPR、LDS 资源共同约束。模拟器按以下公式计算每个 CU 能同时容纳的 wave 数（occupancy）：max_waves = min(max_waves_per_simd × num_simd, total_vgpr / vgpr_per_wave, total_sgpr / sgpr_per_wave, total_lds / lds_per_workgroup × waves_per_workgroup)。Occupancy 直接影响延迟隐藏能力：更多 wave 意味着长延迟操作期间有更多 ready wave 可供切换。

3.7 内存子系统模型

3.7.1 地址空间

AMDGPU 有多个地址空间：Global（全局显存）、Local（LDS）、Private（scratch/stack，per-lane）、Constant。功能仿真中，Global 和 Constant 映射到 host 侧 mmap 内存，Local 映射到 workgroup 粒度分配的 LDS buffer（host 侧 malloc），Private 映射到 per-wave 的 scratch buffer。

3.7.2 Cache 延迟模型

模拟器提供两种可切换的 cache 模型。固定延迟模型（默认）：所有 VMEM 请求统一使用配置文件中指定的延迟值（如 L1_hit=40, L2_hit=120, HBM=300），用户可根据预期的 cache 命中率手动设置平均延迟。概率模型（可选）：根据访问模式估算 L1/L2 命中概率，对每个请求随机采样决定延迟。参数为 L1_hit_rate 和 L2_hit_rate，实际延迟 = L1_hit_rate × L1_latency + (1-L1_hit_rate) × L2_hit_rate × L2_latency + ... 以此类推。

3.7.3 Outstanding 请求跟踪

每个 wave 维护三个 outstanding 计数器：vmcnt（VMEM 请求数）、lgkmcnt（LDS + SMEM 请求数）、expcnt（export 请求数）。当发射 VMEM load 时 vmcnt++，当模拟的延迟 cycle 到达时 vmcnt--。s_waitcnt vmcnt(N) 表示 wave 必须等到 vmcnt ≤ N 才能继续。模拟器在时间轴上为每个 outstanding 请求记录预期完成 cycle，当 s_waitcnt 时检查是否已满足条件。

3.7.4 带宽模型

简化带宽建模方式：为每个内存层级维护一个 "tokens per cycle" 计数器（如 HBM 带宽对应每 cycle 可服务的 bytes/CL 数）。当某 cycle 的 outstanding 请求超过 token 限额时，超出的请求被推迟到后续 cycle（延迟增加）。这比精确的 NoC 仿真轻量得多，但足以反映带宽瓶颈。

3.8 同步原语与屏障

3.8.1 s_barrier

当 wave 执行 s_barrier 时进入 BARRIER_WAIT 状态。每个 workgroup 维护一个 barrier counter，记录已到达 barrier 的 wave 数。当 counter == workgroup 内 wave 总数时，所有 wave 同时切换回 READY 状态，barrier cycle 为最后一个 wave 到达的 cycle。被 barrier 阻塞的 cycle 记入 BARRIER_STALL bubble 统计。

3.8.2 LDS 原子操作与数据竞争

对于 ds_read/ds_write 等 LDS 操作，功能仿真保证 wave 内各 lane 按 lane ID 顺序执行（模拟硬件的序列化行为）。对于跨 wave 的 LDS 原子操作（如 ds_add_u32），因为 wave 是 lock-step 执行，不同 wave 之间的操作顺序由调度器决定，模拟器保证同一 workgroup 内的 LDS 操作在 barrier 之间是确定性的（给定调度策略下）。

3.9 Grid 级调度与多 CU 模型

3.9.1 Workgroup 到 CU 的分配

Grid 级调度器维护一个 workgroup 队列（总数 = gridDim.x × gridDim.y × gridDim.z / blockDim）。每个模拟 cycle 检查是否有 CU 有空闲资源容纳新 workgroup。分配策略为 round-robin 或 fill-first（优先填满一个 CU 再用下一个）。当 workgroup 被分配到 CU 时，在该 CU 上创建对应数量的 wave 并初始化其 thread ID、workgroup ID、kernel arg pointer 等寄存器。

3.9.2 全局 Cycle 计算

关键设计点：输出的不是单个 wave 的 cycle，而是整个 kernel 的执行 cycle。kernel_total_cycle = 最后一个 wave 在所有 CU 上完成执行的 cycle - kernel 开始的 cycle。由于不同 CU 上的 workgroup 并行执行，全局 cycle 取所有 CU 中最大的完成 cycle。多个 kernel launch 的程序总 cycle = Σ kernel_total_cycle（当前不考虑 kernel 重叠）。

3.9.3 多 CU 并行模拟策略

模拟器自身在 host CPU 上是单线程或少量线程运行。两种策略可选。锁步模拟：维护全局 cycle counter，每 cycle 推进所有 CU。保证时序精确但较慢。独立推进 + 同步点：各 CU 独立推进自己的 cycle，仅在需要跨 CU 交互时（如 global memory 原子操作、L2 cache 共享）同步。对于大多数 compute kernel（CU 间无数据依赖），此策略可显著加速模拟。默认使用独立推进策略。

3.11.1 统计数据采集

struct BubbleStats {

3.11.2 Bound 归因算法

在全部模拟完成后，性能分析引擎根据汇总的 bubble 统计数据计算瓶颈归因。算法如下：首先计算各类 bubble 占总 cycle 的百分比。然后判定主瓶颈（dominant bound）：若 valu_active_cycles + mfma_active_cycles 占 total_cycles 的比例 > 60%，且 bubble 中 no_ready_wave 占比小，判定为 Compute Bound。若 bubble_waitcnt 占比最大且 vmem_requests 量大，判定为 Memory Bound（Global）。若 bubble_waitcnt 占比最大且 lgkm（LDS+SMEM）请求量大，判定为 LDS/SMEM Bound。若 bubble_no_ready_wave 占比最大（意味着所有 wave 都在等待长延迟操作），判定为 Latency Bound（occupancy 不足）。若 bubble_barrier 占比显著，判定为 Barrier Bound（workgroup 内负载不均）。

        const uint8_t* code, size_t code_size,
        GfxTarget target);
    
    // 解码单条指令 (用于单步调试)
    InsnObj decodeSingle(
        const uint8_t* pc_ptr, GfxTarget target);
};

// ========== Trace Writer 接口 ==========

public:
    void setLevel(TraceLevel level);
    void setFilter(const TraceFilter& filter);
    void recordInsn(const TraceRecord& record);
    void flush();
    void close();
};

6.3 Perfetto 时间线

如 3.12 节所述，生成 JSON 或 protobuf 格式的 trace 文件。用户可直接在浏览器中打开 ui.perfetto.dev，拖入 trace 文件进行交互式分析。关键可观察信息包括：每个 CU 每个 SIMD 上 wave 的执行时间线（色块表示指令类型，灰色表示 bubble）；鼠标悬停查看指令详情（PC、助记符、寄存器值）；搜索特定 PC 地址定位所有相关 event；counter track 展示 IPC、occupancy、outstanding VMEM 请求数等随 cycle 变化的曲线；flow event 将 VMEM 请求发射与数据返回关联起来，可视化内存延迟。

使用方式： 运行模拟后在终端执行 python3 tools/open_perfetto.py gpusim_perfetto.json，脚本会自动启动浏览器打开 Perfetto UI 并加载 trace。也可手动拖入 ui.perfetto.dev。

7. 验证与校准策略

7.1 功能正确性验证

功能正确性是第一优先级。验证策略分三个层次。指令级单元测试：为每种 ISA 指令编写单元测试，输入已知的寄存器值和内存状态，验证执行后的结果与 ISA 手册描述一致。覆盖边界情况（NaN、denorm、overflow、EXEC mask 部分为零等）。Kernel 级对比测试：编写一组覆盖各种 pattern 的 HIP kernel（GEMM、reduction、scan、transpose、softmax 等），分别在真实 GPU 和模拟器上运行，比较输出结果（允许浮点 ULP 容差）。端到端应用测试：用真实的训练或推理脚本（如 PyTorch 模型）运行在模拟器上，验证最终结果。

7.2 Cycle 精度校准

Cycle 模型不追求绝对精度，但需要保证"相对排序正确性"——即优化前 kernel A 比 kernel B 慢，在模拟器中也应体现出 A 更慢。校准方法包括：选取 10-20 个代表性 kernel，分别在真实 GPU 上用 rocprof 测量实际执行时间（cycle 数），与模拟器输出对比。计算相关系数（目标 > 0.9）和平均相对误差（目标 < 30%）。根据偏差调整延迟表参数。定期更新延迟表以跟踪新 GPU 型号。

7.3 回归测试

每次代码变更后自动运行功能正确性测试套件和 cycle 校准 benchmark。在 CI 中设置功能测试 pass 率 100% 的 gate 和 cycle 相关系数 > 阈值的 warning。

8. 实现路线图

Phase 1：功能仿真 MVP 4-6 周

实现 Runtime 拦截层（hipMalloc/Free/Memcpy/Launch/Sync）。实现 ELF 解析器，提取 kernel .text 和 metadata。实现指令解码引擎（优先覆盖 SOP1/2, VOP1/2/3, SMEM, MUBUF, DS, SOPP 等高频格式）。实现功能仿真核心（WaveState, VGPR/SGPR 读写, EXEC mask, 基本分支, LDS, barrier）。目标：能够正确运行 vector_add, reduction 等简单 kernel 并通过结果校验。此阶段无 cycle 模型。

Phase 2：Cycle 模型 + 基础统计 4-6 周

实现 CU Timing Model（scoreboard, 功能部件 ready cycle, 发射判定）。实现 Wave Scheduler（round-robin）。实现简化内存延迟模型（固定延迟）。实现 Grid 级调度器。实现 BubbleStats 采集。目标：输出每个 kernel 的总 cycle、bubble 分类、dominant bound 判定。终端摘要报告。

Phase 3：Trace / Log / Perfetto 3-4 周

实现 TraceWriter（text + binary 格式，多级别，过滤机制）。实现 PerfettoWriter（JSON Chrome Trace Format）。实现 JSON 统计报告输出。工具脚本（trace 浏览、Perfetto 打开）。目标：完整的可视化分析能力。

Phase 4：指令集扩展 + MFMA + 精度提升 4-6 周

扩展指令集覆盖（MFMA 全系列、FLAT、GLOBAL、SCRATCH、VOP3P 等）。实现 MFMA 功能仿真和 timing。引入概率化 cache 模型。LDS bank conflict 建模。支持 wave32 模式。目标：能够正确模拟 GEMM、FlashAttention 等复杂算子。

Phase 5：生产化 + 校准 3-4 周

多 GPU target 配置预设（MI100/MI210/MI300X）。Cycle 校准（对标真实硬件 rocprof 数据）。CI 回归测试框架。性能优化（模拟器自身的运行速度）。文档和使用指南。Perfetto protobuf 格式升级。

总估计工期：18-26 周（约 4.5-6.5 个月），按 2-3 人团队。Phase 1-2 完成后即可开始为编译器团队提供基本分析能力，后续 phase 可并行于实际使用中迭代。
