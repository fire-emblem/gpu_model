# LLVM AMDGPU Integration Roadmap

> [!NOTE]
> 历史计划文档。用于保留当时的拆解和决策上下文，不作为当前代码结构的权威描述。当前主线以 `docs/my_design.md` 和 `docs/module-development-status.md` 为准。


## Objective

The next major step after the current basic ISA subset is to consume instructions
produced by a real LLVM AMDGPU backend, rather than relying only on hand-built
instruction streams.

The goal is not to embed a full compiler inside this project.
The goal is to let the simulator ingest real AMDGPU-oriented compiler output so that:

- compiler codegen changes can be evaluated on realistic instruction streams
- operator-library kernels can be compared using generated machine-like code
- the simulator remains a lightweight analysis tool rather than a full toolchain fork

## Current Environment Constraint

The local environment now provides:

- `clang`
- `llc`
- `llvm-mc`
- `hipcc`
- `clang-offload-bundler`
- `llvm-objcopy`

That means this project can already:

- emit real AMDGPU assembly from LLVM IR
- compile minimal HIP host artifacts that embed `.hip_fatbin`
- extract an `amdgcn-amd-amdhsa` bundle from those HIP artifacts
- load the extracted device code object into the existing object-loader path

What is still missing is **ISA coverage**, not external toolchain presence.
Real HIP kernels can now be parsed out of host artifacts, but only kernels whose
disassembly stays within the simulator's supported AMD-style instruction subset
are expected to execute successfully.

## Recommended Overall Direction

Use the LLVM AMDGPU backend as an **external producer** and this project as an
**artifact consumer**.

Do **not** make the simulator depend on LLVM libraries in the first version.

The right layering is:

1. external compiler toolchain
2. exported AMDGPU artifact
3. project-side loader / parser / lowerer
4. `KernelProgram`
5. functional / cycle executors

## Recommended Phases

### Phase 1: Real AMDGPU Assembly Ingestion

Target:

- accept AMDGPU-style assembly text generated by LLVM backend tools

Recommended artifact:

- textual AMDGPU assembly

Why this first:

- easiest to inspect
- easiest to diff
- easiest to debug unsupported instructions
- does not require immediate ELF/code-object fidelity

Project additions needed:

- a dedicated AMDGPU assembly loader, separate from the current simplified asm parser
- a dialect-aware parser for real AMDGPU mnemonic forms
- a lowering layer:
  - `AMDGPUAsm -> KernelProgram`
- explicit unsupported-instruction diagnostics

Important rule:

- do not try to parse every AMDGPU instruction at once
- support only the minimal subset seen in target kernels first

### Phase 2: Metadata And Code Object Awareness

Target:

- preserve launch-critical metadata from real compiler output

Relevant information includes:

- kernel entry name
- SGPR/VGPR usage
- LDS usage
- kernarg layout
- target architecture / gfx level
- workgroup requirements

Project additions needed:

- code-object metadata reader or sidecar metadata reader
- mapping from AMDGPU metadata to the existing `KernelMetadata` / `LaunchRequest` model

Important rule:

- metadata fidelity matters more than full binary fidelity in the first step

### Phase 3: Optional Binary / HSACO Path

Target:

- consume an object or code-object artifact directly

Possible artifact families:

- ELF/code object
- HSACO-like bundles

Project additions needed:

- binary section reader
- extraction of:
  - text / instruction bytes or disassembly source
  - metadata
  - constant sections
  - symbol table information

Important rule:

- if binary decode is added, keep the simulator’s internal execution representation as `KernelProgram`
- do not make the executors operate on raw binary bytes directly

### Phase 4: Optional Toolchain Driver Wrapper

Target:

- add a helper wrapper that invokes LLVM AMDGPU compilation externally when available

This wrapper should:

- detect toolchain availability
- invoke external compiler commands
- capture stdout/stderr
- materialize intermediate artifacts for debugging

It should **not** become a mandatory runtime dependency for the simulator core.

## Recommended Internal Architecture

### New Logical Layers

1. `compiler_artifact`

Suggested purpose:

- represent outputs produced by external AMDGPU compilation

Suggested content:

- assembly text
- metadata blobs
- constant sections
- optional binary/code-object sections
- target arch string

2. `amdgpu_loader`

Suggested purpose:

- load external AMDGPU artifacts into project-internal structures

Suggested responsibilities:

- parse AMDGPU assembly dialect
- validate supported subset
- collect unsupported instructions
- lower to `KernelProgram`

3. `amdgpu_lowering`

Suggested purpose:

- isolate mapping from real AMDGPU syntax into project ISA semantics

Suggested responsibilities:

- map supported mnemonics to existing opcodes
- map unresolved instructions to:
  - explicit unsupported diagnostics
  - or a future generic “external opcode” form if needed

## What To Keep Stable

The following core abstractions should remain stable:

- `KernelProgram`
- `Semantics -> OpPlan`
- `FunctionalExecutor`
- `CycleExecutor`

This keeps LLVM integration as a front-end / loader problem rather than forcing a rewrite
of the execution core.

## What Not To Do First

Avoid these in the first LLVM/AMDGPU step:

- linking directly against full LLVM C++ APIs
- decoding the full AMDGPU binary ISA stream immediately
- supporting every GCN/RDNA mnemonic family at once
- mixing simulator execution semantics with compiler driver logic
- requiring a local compiler toolchain just to build or run the simulator

## Minimal First Supported Instruction Families

The first real LLVM-backed subset should focus on instructions already close to what
the simulator supports:

- scalar ALU / compare / branch
- `s_waitcnt`
- `s_buffer_*`
- `buffer_*`
- `ds_*`
- simple `v_*` integer ALU and compare
- mask / exec control instructions

This gives the best return while minimizing parser/lowering complexity.

## Success Criteria

This integration is successful when:

- a real LLVM-produced AMDGPU assembly artifact can be lowered into `KernelProgram`
- unsupported instructions are reported clearly
- the resulting kernel can run in both functional and cycle modes
- codegen differences from compiler changes produce measurable cycle differences
- the simulator still remains lightweight and easy to debug
